# AWS Non-Lock-In + Postgres Migration Plan

## TL;DR

> **Quick Summary**: Move YUMMY from Cloudflare-oriented Workers/D1/OpenNext packaging to a practical AWS architecture using standard Docker/OCI images, ECS Fargate, ALB, RDS PostgreSQL, and OpenTofu/Terraform-compatible IaC. Preserve portability by keeping runtime surfaces standard: Node, containers, HTTP, Postgres, and HCL IaC.
>
> **Deliverables**:
> - Backend migrated from SQLite/D1/BetterSQLite3 to RDS-compatible PostgreSQL via Drizzle pg-core.
> - Frontend migrated from OpenNext Cloudflare Worker packaging to standard Next.js standalone/container deployment.
> - Cloudflare runtime/tooling references removed or explicitly marked legacy and excluded from AWS path.
> - Local Docker parity: frontend + backend + Postgres run without Cloudflare.
> - OpenTofu AWS dev environment: VPC, ECR, ALB, ECS Fargate services, RDS PostgreSQL, Secrets Manager.
> - GitHub Actions CI/CD builds, tests, pushes images, deploys dev, and runs smoke QA.
>
> **Estimated Effort**: Large
> **Parallel Execution**: YES - 4 waves + final verification
> **Critical Path**: T1 → T5 → T6/T7 → T8/T11 → T12/T13 → T14 → Final Verification

## Context

### Original Request
The project is currently vendor locked-in to Cloudflare for both backend and frontend. The target is AWS infrastructure without unnecessary vendor lock-in, and the database should move from SQLite to Postgres.

### Interview Summary
**Key Discussions**:
- User selected **Practical AWS** portability: standard containers + RDS Postgres + IaC, not maximum Kubernetes portability.
- User selected **RDS Postgres** over Aurora or self-managed Postgres.
- User requested a **work plan** rather than only a research memo.

**Research Findings**:
- Frontend Cloudflare coupling exists in `frontend/package.json` and may be regenerated by OpenNext Cloudflare builds as `.open-next/**` Worker artifacts containing Cloudflare image bindings (`ASSETS`, `IMAGES`), Durable Object-generated artifacts, and skew-protection code. Current checkout does not contain `frontend/.open-next/**`; tasks must inventory/remove generated artifacts if present and prevent AWS CI from generating them.
- Backend Cloudflare coupling exists in `backend-ts/package.json`, `backend-ts/wrangler.jsonc`, `backend-ts/src/db/client.ts`, and `backend-ts/src/app.ts` through D1/Worker `c.env?.DB` wiring.
- Backend already has a useful Node entrypoint at `backend-ts/src/index.ts` using `@hono/node-server`.
- Existing AWS docs/configs are present: `docs/aws/APP_RUNNER_AMPLIFY.md` and `apprunner.yaml`; App Runner docs should be treated as legacy/reference, not target architecture. `docs/aws/AWS_DEPLOYMENT_ECS_FARGATE.md` and `docs/aws/CI_CD_GITHUB_ACTIONS.md` do not exist yet and must be created by this plan.
- Database migration touches `backend-ts/drizzle.config.ts`, `backend-ts/src/db/client.ts`, `backend-ts/src/db/client.local.ts`, `backend-ts/src/db/migrate.ts`, `backend-ts/src/db/schema.ts`, `backend-ts/src/db/migrations/*`, repository files, and `backend-ts/tests/integration/_setup.ts`.

### Metis Review
**Identified Gaps** (addressed):
- Need explicit SSE/ALB timeout verification because the backend uses streaming/SSE flows.
- Need clear scope boundaries: no dependency upgrades, route refactors, auth changes, Redis, queues, WAF, multi-region, Aurora, or EKS.
- Need pre-migration baseline before changing runtime or DB.
- Need Docker Postgres test strategy, not only pg-mem.
- Need explicit treatment of SQLite JSON/boolean/datetime/id semantics.
- Need AWS deployment as its own verifiable phase, not an infra side quest.

### Assumptions Applied
- Frontend requires SSR/full Next.js support unless executor proves static export is sufficient during inventory.
- Single AWS dev environment is enough for this plan; staging/prod rollout is out of scope.
- Existing production data volume is unknown, so the plan includes a data export/import path and a dry-run migration.
- OpenTofu is the default IaC tool because it maximizes practical portability while retaining Terraform compatibility.
- Docker Postgres is the default integration-test strategy; pg-mem is fallback only.

### Risk Flags
- **HIGH**: SSE compatibility under ALB and Fargate must be verified with long-lived `curl -N` scenarios.
- **HIGH**: Drizzle migration history cannot be reused directly from SQLite; Postgres migrations must be regenerated.
- **MEDIUM**: Durable Object usage is generated and must be verified before declaring Cloudflare removal complete.
- **MEDIUM**: Frontend SSR/static-export assumption affects hosting complexity.
- **LOW**: Mixed package managers (`pnpm` backend, `npm` frontend) add Docker/CI complexity.

## Work Objectives

### Core Objective
Produce a working AWS dev deployment for YUMMY that runs without Cloudflare runtime dependencies, uses RDS PostgreSQL instead of SQLite/D1, and remains portable through standard containers, Postgres, and OpenTofu-managed infrastructure.

### Concrete Deliverables
- Postgres-backed backend using Drizzle pg-core and standard Node runtime.
- Standard Next.js standalone/container frontend build.
- Local Docker Compose parity environment with Postgres.
- OpenTofu modules for AWS dev infrastructure.
- GitHub Actions CI/CD for build/test/deploy/smoke.
- Updated docs marking legacy Cloudflare/App Runner paths and documenting AWS ECS Fargate path.

### Definition of Done
- [ ] Backend tests pass against Postgres, not SQLite `:memory:`.
- [ ] Frontend build/tests pass without `@opennextjs/cloudflare` build path.
- [ ] Local Docker Compose starts Postgres + backend + frontend and serves health/API/UI responses.
- [ ] OpenTofu `init`, `validate`, and `plan` pass for dev.
- [ ] AWS dev ECS services reach desired running count and pass ALB health checks.
- [ ] RDS migration dry-run succeeds and data shape is validated.
- [ ] No runtime AWS path depends on `wrangler`, D1, Worker `c.env.DB`, Cloudflare image bindings, or generated `.open-next` Worker runtime.
- [ ] All final verification agents approve and user gives explicit okay.

### Must Have
- RDS PostgreSQL as production database target.
- ECS Fargate + ALB as compute target.
- OpenTofu/Terraform-compatible IaC.
- Standard OCI images for frontend and backend.
- Preservation of existing API shapes, especially snake_case schemas.
- Agent-executed QA for every task.

### Must NOT Have (Guardrails)
- No Aurora, EKS, Lambda rewrite, App Runner primary target, WAF, multi-region, Redis/ElastiCache, SQS, or application DynamoDB unless a verified active Cloudflare primitive requires explicit fallback planning. **Exception**: DynamoDB is allowed only as OpenTofu remote-state lock table infrastructure, not as an application datastore.
- No business logic refactors, route shape changes, auth changes, schema field renames, or dependency major upgrades.
- No auto-camelCase conversion of API schemas.
- No copying `.env` files into container images.
- No running containers as root.
- No line-by-line reuse of SQLite migration SQL for Postgres.
- No human-only acceptance criteria.
